#+title:     Longevity Pantry
#+language:  en

* Introduction

Longevity Pantry is a small, browser-based app for managing a longevity-oriented
kitchen.  It runs locally as a personal, single-user tool, and can also be
hosted with per-user accounts.  Its code, crates, and configuration keep the
original =food4u= name for convenience.

The page is four side-by-side columns, each collapsible to a vertical bar:

- *Longevity Foods* — a food pyramid (Foundation → Occasionally) of ~500 whole
  foods, organized by category and drawn from Mediterranean, Okinawan, Nordic,
  and Blue Zones patterns.
- *Recipes* — your recipes, grouped by meal category.  Add one by name or paste
  a recipe as text and it is parsed into ingredients and instructions.  Filter
  to "Can make now" or "Almost there" based on what is stocked.
- *Kitchen* — storage panes (Pantry, Refrigerator, Freezer, Counter, Spice
  Cupboard, Apothecary) tracking what you actually have on hand.
- *Shopping List* — what to buy, grouped by grocery-store department, with a
  one-click plain-text export.

You build it by dragging foods between columns: pyramid → kitchen or shopping
list (copy), recipe ingredients from the pyramid, and a recipe's missing
ingredients to the shopping list in one click.  Recipes can be linked to pyramid
foods, and a recipe is flagged green when every ingredient is already stocked.

The frontend is an [[https://elm-lang.org/][Elm]] single-page app served by a small Rust server (built on
the rust-template foundation).  The server stores each user's model in a
relational database — SQLite for a local install, PostgreSQL for a hosted one.
Edits in the browser save automatically.

* Running it

The development entry point builds the frontend, starts the server, and opens
the app in a browser:

#+begin_src sh :exports code
just dev
#+end_src

This serves the app at =http://127.0.0.1:3000=.  When no OIDC provider is
configured the server runs unauthenticated, so there is no login step for local
use.  To try the signed-in, multi-account experience locally, =just login-demo=
stands up a local identity provider alongside the server; see
[[file:dev/oidc/login-demo.org][=dev/oidc/login-demo.org=]].

Run everything through the Nix dev shell so the Rust, Elm, and formatter
toolchains match:

#+begin_src sh :exports code
nix develop --command just dev
nix develop --command just build
nix develop --command just test
#+end_src

* Architecture

The frontend is an Elm single-page app, split into a module per column over a
shared core for the model, the messages, and drag-and-drop.  It holds the whole
model in memory and syncs it to the server over a small JSON API.

The Rust server serves the compiled frontend and that API.  It resolves each
request's user from the OIDC session — the signed-in account, or a single local
user when no provider is configured — and reads and writes that user's model in
a relational database behind one typed model and schema: SQLite for a local
install, PostgreSQL for a hosted one.  A new account is seeded the default
storage panes on its first read.

The default model is embedded in the binary, generated from the original Claude
Design mock-up.  A fresh store is populated from it once — along with any legacy
JSON file an earlier version wrote, which is then kept only as a backup — so
nothing is lost in the move to the database.

** Model API

The browser syncs over two JSON endpoints, each scoped to the requesting user:

- =GET /api/model= — that user's model, seeding the default panes on a first
  visit.
- =PUT /api/model= — persist the posted model; it is checked against the typed
  schema and rejected if it exceeds the per-user size limits.

* Configuration

Configuration follows the foundation's staged precedence (CLI flags, then
=food4u_*= environment variables, then a TOML config file, then compiled
defaults).  The app-specific knobs:

- =--database-url= / =food4u_database_url= — the relational store (default
  =sqlite:food4u.db=; a =postgres://= url selects PostgreSQL for a hosted
  deployment).
- =--data-file= / =food4u_data_file= — a legacy JSON model imported once into an
  empty store, then kept only as a backup (default =food4u-model.json=).
- =--frontend-path= / =food4u_frontend_path= — the compiled frontend assets
  (default =frontend/public=).
- =--base-url= / =food4u_base_url= — required; the public base URL.  =just dev=
  sets this to the local address for you.

Authentication is off unless an OIDC provider is configured, through
=food4u_oidc_issuer=, =food4u_oidc_client_id=, and
=food4u_oidc_client_secret_file=.  With it off the server runs as a single local
user; with it on, each account keeps its own kitchen.

* Gotchas and tradeoffs

- The server understands the model: typed Rust structs mirror the Elm schema and
  a relational schema stores it, so a malformed document is rejected at the edge
  rather than persisted.  The frontend still owns the schema's shape, and the
  two definitions are kept in step by hand.
- The relational store (=food4u.db= for SQLite) is the store of record and is
  git-ignored, so edits survive rebuilds.  A fresh store is populated once from
  the embedded default; delete the file to start over from it.
- The SQLite backend is the verified path; the PostgreSQL backend that a hosted
  deployment uses is written but not yet exercised against a live server in CI.
- Drag-and-drop uses native HTML5 DnD.  A capture-phase listener in =index.html=
  seeds =dataTransfer= so drags initiate in Firefox and Safari as well as
  Chrome.
