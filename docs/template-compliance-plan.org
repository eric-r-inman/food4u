* The Windows exception

This is the only place where "what the template emits today" and "what food4u
can ship" genuinely conflict, so it gets a full accounting.

** What the template now requires

The template added an always-on Windows cross-compile: =template/flake.nix=
wires =mkWindowsCrossPackages= / =mkWindowsMsvcCrossPackages= and a
=windowsSmoke= check, and three new checks enforce it:

- =flake-windows-x86-package= — a =<crate>-x86_64-windows= package output
  resolves.
- =flake-windows-aarch64-package= — a =<crate>-aarch64-windows= package output
  resolves.
- =flake-windows-smoke-check= — the =windowsSmoke= check exists on x86_64-linux.

The first two are =flake-output-present= checks: they *=nix eval=* the outputs
(prove they resolve), they do not build them.

** Why food4u currently fails two of the three

Earlier this session the Windows cross build was wired in, and CI's "Windows
smoke (wine)" job — which *builds*, not just evaluates — failed:

#+begin_example
error: couldn't read `crates/server/src/seed/seed.sql`: No such file or directory
  --> crates/server/src/seed.rs:16 (const SEED_SQL = include_str!("seed/seed.sql"))
#+end_example

Root cause: the foundation's cross helpers set =src = craneLib.cleanCargoSource
self=, which re-filters the source through crane's default rules and *drops the
=.sql= seed the server embeds with =include_str!=*.  Our flake's own
=commonArgs.src= keeps =.sql= (that fix landed earlier for the native build),
but the helper overrides =src= and never consults =commonArgs.src=, so the
embedded seed vanishes only on the cross paths.  The same latent breakage
affects the darwin / musl / gnu cross packages; nothing builds those today
(=release-binary: false=), so only the Windows smoke job forced a cross build
and exposed it.

To make CI green, the Windows cross packages were removed and the required
=windowsSmoke= check kept over an empty set, where it passes trivially — the
behaviour that check is explicitly designed for "when a project ships no Windows
binaries".  That is why =flake-windows-smoke-check= passes but
=flake-windows-x86-package= / =flake-windows-aarch64-package= now fail.

The bind: re-adding the package outputs satisfies the two eval-only *checker*
rules, but the =windowsSmoke= *CI* job then builds them and fails on the dropped
seed again.  With the foundation helper as-is, the checker and CI cannot both be
green.

** Two ways forward (recommended first)

*Option A — fix the foundation upstream (the fully-compliant path).* Propose a
change to =rust-template= so the cross helpers preserve embedded non-Rust
sources — either by honoring the caller's =commonArgs.src= instead of overriding
it, or by accepting a source-filter argument that keeps =.sql= (and by extension
any =include_str!=-embedded asset).  This fixes a genuine template bug that hits
*any* seed-embedding spawn, not just food4u.  Once the foundation carries it, we
re-add the Windows wiring to our flake and all three checks *and* the CI smoke
job go green — full compliance, no opt-out.  Because our future host authored
the template, this is the change most in the spirit of "be as compliant as
possible": it removes the exception rather than declaring one.  It is also the
only option that lets the darwin / musl / gnu cross builds actually build here.

*Option B — declare the deviation honestly (the interim/fallback).* Until the
foundation carries the fix, list the two package checks under
=compliance-ignores= in =rust-template.json= with a stated reason, keeping the
trivial =windowsSmoke= check so its rule still passes:

#+begin_src json :exports code
"compliance-ignores": [
  { "id": "flake-windows-x86-package",
    "reason": "The server embeds seed.sql via include_str!; the foundation cross helper drops it, so the Windows PE cannot build. A hosted Linux service also ships no Windows binary. Tracked for the upstream foundation fix; see docs/template-compliance-plan.org." },
  { "id": "flake-windows-aarch64-package",
    "reason": "Same as flake-windows-x86-package." }
]
#+end_src

An ignored check reports as ignored, not failed, and does not affect the exit
code — the checker stays honest about a declared, reasoned deviation.  If we
keep Windows off, also ignore =contributing-windows-binaries-section= for the
same reason rather than documenting binaries we do not build.

*Recommendation.* Pursue Option A (open the foundation PR) as the real fix, and
apply Option B in the same compliance commit as the honest interim so CI and the
checker are both green today.  Revert the ignores once the foundation fix is in.

* README.org — a difference, not a gap

The user asked how our README.org compares with the template's.  The template
=README.org= is a *fill-in-the-blanks scaffold* ("=#+title: <project-name>=",
"Briefly describe the problem this project solves…", "Explain how to install…").
Ours is a *fully written project README* describing the four-column app, how to
run it (=just dev=, =just login-demo=), and the Elm-front / Rust-server / SQLite
or Postgres architecture.

This is expected and correct: =docs/compliance.org= states plainly that "the
content of README.org … is for the human" and is *not* a compliance item.
Replacing the scaffold with real content is exactly its intended use, so there
is nothing to reconcile.  One cosmetic nit, not a compliance item: our README's
=#+title:= line is empty; consider setting it to "Longevity Pantry".
