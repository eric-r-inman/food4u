* Server crate: role and scope

The =crates/server= crate is the project's general-purpose long-running service.
It is /not/ specifically an HTTP server — HTTP is present only as infrastructure
for health checks, metrics, and observability.

The server might primarily:

- Watch local files for changes and react to them.
- Communicate over a binary protocol, gRPC, or WebSockets.
- Bridge between systems (queues, databases, external APIs).
- Serve any other role that calls for a persistent process responding to
  stimuli.

When adding new functionality that requires a long-running process, build it in
the server crate rather than creating a separate service.  The server already
provides structured logging, systemd socket activation, watchdog integration,
graceful shutdown, and an Elm frontend — duplicating this infrastructure is
wasteful and error-prone.

Do not mistake the server for "just a web server" and stand up a separate HTTP
service from scratch.  The existing HTTP routes (=/healthz=, =/metrics=, =/me=,
=/scalar=) are supporting infrastructure, not the defining purpose.

* Persistence and storage

The food model's store of record is a relational database — SQLite for local
development, Postgres for the hosted deployment, chosen by the ~database_url~
scheme.  All reads and writes go through the repository (=repo.rs= for SQLite,
=repo_pg.rs= for Postgres) behind the ~Db~ enum.

There is no JSON-file persistence, and none is to be added.  Do not read or
write the model — or any new server-side state, such as settings or sessions —
as a JSON file on disk; new persisted data gets a relational schema and a
migration.  The JSON model file the server once wrote, and the one-time import
step that read it, are gone.

Only the HTTP API's request and response bodies are JSON — that is the transport
between the Elm frontend and the server, not storage.  Nothing on disk is: even
the bundled defaults are a portable SQL fixture (=seed.sql=) the server runs
into an empty database on first start, edited through the app (see
=docs/editing-default-data.org=), not a JSON document.

* General code guidelines

These are instructions for LLMs to write code more to my liking.

** Documents

When authoring new content, make them ~org-mode~ files.

Use ~:exports code~ if you just want to show off the code, and don't want it to
actually run anything.

No file lists in documentation.

** Comments

*** Formatting and punctuation

Comments must be complete sentences, wrapped at ~80 columns~, with standard
punctuation.

*** Content

Only comment non-obvious intent, invariants, tradeoffs, or historical
constraints; no restating of control flow.

** Version Control
*** Commit Messages

A commit message logs a change that is suitable for a reviewer.  Generally
speaking, we want to submit one commit per pull request.  Multiple commits can
be presented in a pull request if somehow the changes are completely different
yet required in order for the pull request to be considered complete.  Many git
forge sites default the one-and-only-one commit as both a pull request title and
description (from the commit summary and body).  We want to leverage that as
much as we can.

For the commit message content itself, the summary should be present tense with
no prefixes and no "Conventional Commit" tagging.  Just tell us what happens if
we apply the commit.  You can imagine a sentence starting with "Applying this
commit will ..." an the commit message summary completes the sentence.  No
punctuation at the end (this is our one and only one exception to the
punctuation rule).

Good:

#+begin_example
clarify documentation re: ritual summoning
#+end_example

#+begin_example
simplify approach of detecting tachyons
#+end_example

#+begin_example
add flux capacitor to front-end
#+end_example

Bad:

#+begin_example
(feat) Add a flux capacitor to the front-end in main.elm
#+end_example

The commit body is your opportunity to document the motivation for the changes
(e.g. to fix a bug, to optimize a workflow, to add a new feature, and so on).
This is the "why" part.  Why did were these changes necessary?  In addition,
provide context for the change.  Does the bug fix come at some cost to
performance?  Were other considerations made as part of the changes, and then
discarded?  Is this the first commit in a series of similar refactors?

A commit body that contains line numbers, file names, and symbol names (e.g.
functions, variables, etc.) is a high scoring smell and should be avoided.

One thing to be mindful of when crafting a commit is that whoever is viewing the
commit _can always see what changed for themselves_.

If you find yourself including critical information in the commit whose value
extends outside of the commit (e.g. that this is a workaround or some surprising
change), that should exist as a code comment.  It can be worth calling out such
workarounds in the commit as well, but you don't need to specifically locate
them (no line numbers, function names, and so on).

Another reason not to include line numbers is that code formatters will change
this as part of making the commit.

*** History Hygiene

We want to avoid commits that are essentially "I accomplished a task" and then
later a "jk here's the rest of the task done".  When this happens, we should
perform a fixup assuming the commits are unpublished.

This tends to come up especially when you have a mega-commit for a bunch of
unrelated work.  In these cases, the right thing to do is break up the work
assuming the work can survive independently.

In the event that you can't resolve these notions, halt and ask the operator for
direction.

** Rust

*** Error Handling

Prefer semantic errors.  Do not just wrap or propagate a ~FileMissingError~ or
~FileWriteError~.  This information _might_ include what file was involved, but
it doesn't say what was happening.  Include something either about the error
itself (like ~ConfigFileMissingError~) or about the process in which the error
occurred (~ReportFileWriteError~).

In cases where you can be semantic with your approach to APIs, use ~thiserror~
to automatically map API errors of certain kinds to semantic errors.

A bad example.  This is bad because it simply signals that a file somewhere in
the program couldn't be read — this could be any file, and tells contributors
and users nothing:

#+begin_src rustic :results none :exports code
use std::fs;
use thiserror::Error;

enum AppError {
  #[error("Could not read file: {0}")]
  FileReadError(#[from] std::io::Error),
}

fn config_file_contents() -> Result<String, AppError> {
  fs::read_to_string(file_path)
    .map_err(AppError::FileReadError)
}
#+end_src

The following is a good example, where the error indicates the nature of the
failure (the file could not be read) and the context (which file/operation):

#+begin_src rustic :results none :exports code
use std::fs;
use thiserror::Error;

enum AppError {
  #[error("Could not read configuration file: {0}")]
  ConfigFileReadError(#[from] std::io::Error),
}

fn config_file_contents() -> Result<String, AppError> {
  fs::read_to_string(file_path)
    .map_err(AppError::ConfigFileReadError)
}
#+end_src

Generally this means there will be little reuse of error variants, and that is
acceptable. Do not swallow errors — let them propagate.

*** No ~unwrap~ or ~expect~ in non-test code

~.unwrap()~ and ~.expect()~ (the panicking variants — /not/ ~unwrap_or~,
~unwrap_or_else~, or ~unwrap_or_default~, which are fine) are prohibited in all
non-test code.  No carve-outs.

Errors must never be silently swallowed either: no ~let _ = ...~ on a ~Result~,
no ~.ok()~ to discard, no empty match arm dropping the ~Err~.  Propagate with
~?~, transform with ~.map_err(...)~, or handle explicitly.

Tests are the only exemption.  Code under ~#[cfg(test)]~, ~#[test]~, and
integration tests under ~tests/~ may use ~.unwrap()~ and ~.expect()~ freely —
panicking is the desired failure signal there, and tests are not shipping code
paths.

**** The principle

~Result~ is not primarily a recovery mechanism.  Its primary value is /typed
control flow and semantic stamping/ — it forces the caller to acknowledge a
failure mode and lets you propagate, transform, or convert errors in a linear,
reasonable-about way.  Recovery is one thing you can do with that; propagation
and forced acknowledgment are the rest.

~.unwrap()~ and ~.expect()~ discard that value in exchange for cosmetic brevity.
That trade is never worth it in shipping code.

If a failure is genuinely impossible, /prove it to the compiler/ via a macro,
type, or ~const~.  Do not assert it to a future reader with ~.unwrap()~.

**** Compile-time-known literals: use a macro

The most tempting exception is "but the input is a string literal, it can only
fail on a programmer typo."  The fix is not to allow ~.unwrap()~ — it is to move
the validation to compile time:

- *Regex.* Use ~lazy_regex::regex!("^foo$")~.  The proc-macro validates the
  pattern at expansion time, returns a ~&'static Regex~, and caches it.  Invalid
  patterns become compile errors.
- *IP addresses.* Do not parse — ~Ipv4Addr::new(127, 0, 0, 1)~ is ~const~.  Same
  for ~Ipv6Addr~ and ~SocketAddrV4~ / ~SocketAddrV6~ construction.
- *URLs.* Use ~url-macro~ or similar; or build with ~const_format~.
- *General pattern.* Any proc-macro can validate a string literal at expansion
  time. The ~.unwrap()~, if any, lives inside macro-generated code on a path the
  proc-macro has statically proven unreachable — never at the call site.

If no suitable macro exists for a given parse, the right move is /write one/,
not allow ~.unwrap()~.

**** Considered exceptions and why each was rejected

These are the cases that look reasonable on first pass.  The argument for each
and the rejection reasoning are captured here so they don't get re-derived
incorrectly.

***** Literal regex / literal parse without a macro

/Argument for:/ "The pattern is a string literal known at compile time; the only
failure mode is a programmer typo, caught the first time the code runs."

/Why rejected:/ "Caught the first time the code runs" assumes 100% line coverage
of every path containing a ~.unwrap()~.  That is fantasy.  A
~Regex::new("...").unwrap()~ on a cold or rare code path is a landmine for the
next person — it ships, and the panic surfaces in production on the unusual
input that finally hits that branch.  If the input really is a literal, prove it
at compile time with a macro (see above).  Do not assert it at runtime.

***** ~std::sync::Mutex~ / ~RwLock~ poisoning

/Argument for:/ "A poisoned lock means another thread panicked
mid-critical-section.  The protected invariants may be broken; recovering is
almost always wrong.  ~.unwrap()~ is idiomatic."

/Why rejected:/ That is an /application author's/ judgment smuggled into a rule
that also governs library code.  A library that panics on poisoning takes the
recovery decision away from a consumer who might be a long-running daemon, a
supervisor process, an FFI host, or something else entirely.  Return ~Result~
and let the caller decide.  If a project standardizes on ~parking_lot::Mutex~,
this whole class of unwrap goes away — no poisoning, no ~.unwrap()~ needed.

***** ~build.rs~ and other build-time scripts

/Argument for:/ "Build scripts run once at build time.  A panic gives a clear
build failure with a backtrace.  ~Result~ plumbing is pure ceremony."

/Why rejected:/ What looks like ceremony is the actual value.  Write ~fn main()
-> Result<(), Box<dyn Error>>~ in the build script and ~?~ through the whole
thing — cargo still sees a non-zero exit and prints the error. You keep typed
control flow, composable error transformation, and you avoid pushing correctness
onto "specific arrangement of code" in the exception-language style.  Otherwise
non-trivial ~build.rs~ drifts toward needing its own unit tests just to verify
error paths, which is a giant step back from what ~Result~ already gives you for
free.

***** Examples and benchmarks

/Argument for:/ "These aren't shipping code paths; relax the rule the same way
tests are exempt."

/Why rejected:/ Unlike tests, examples have no upside from ~.unwrap()~ over ~?~.
Both fail loudly when something goes wrong; ~?~ just composes properly and reads
the same.  More importantly, examples are /teaching material/ — whatever pattern
they show is the pattern users copy.  Modeling ~.unwrap()~ in examples
propagates ~.unwrap()~ everywhere.  Use ~fn main() -> Result<(), Box<dyn
Error>>~ and ~?~ — that teaches the right thing. Benches follow the same logic.

***** ~thread::spawn~ join handle propagating a child panic

/Argument for:/ "~handle.join().unwrap()~ /re-raises/ the child thread's panic
in the parent.  The ~Err~ payload literally is the panic.  Unwrapping is the
correct propagation; not unwrapping would be swallowing it."

/Why rejected, on two counts:/

1. Not panicking is not the same as swallowing.  Rust forces explicit effort to
   swallow an ~Err~ (~let _ =~, ~.ok()~, etc.).  Returning the ~Err~ up the
   stack, or converting it via ~.map_err(...)~ into a typed error, is the
   /opposite/ of swallowing.
2. The framing presumes recovery is the only point of ~Result~.  It is not — see
   the principle above.  You can convert a join ~Err~ (which is ~Box<dyn Any +
   Send>~) into your own error type and propagate it; the caller may genuinely
   want to log it, retry, fail over, or report it structurally rather than die.
   If you specifically want to re-raise a panic, ~std::panic::resume_unwind~ is
   the honest way to say so — it makes the intent explicit instead of disguising
   it as a generic ~.unwrap()~.

***** ~SystemTime::now().duration_since(UNIX_EPOCH)~

/Argument for:/ "Only fails if the system clock is set before 1970, which won't
happen."

/Why rejected:/ It happens.  Misconfigured embedded devices, freshly-imaged VMs
with bad RTCs, containers without time sync.  Use ~.unwrap_or(Duration::ZERO)~
or propagate.

***** "I just checked above" patterns

/Argument for:/ "I called ~if !vec.is_empty()~ on the line above, so
~vec.first().unwrap()~ can't fail."

/Why rejected:/ The check-and-unwrap pair rots the moment someone refactors
between the two lines.  Restructure with ~if let Some(x) = vec.first()~ — the
type system carries the invariant for you, no comment or discipline required.

***** ~env::var("FOO").unwrap()~ for "always-set" env vars

/Argument for:/ "We always set ~FOO~ in our deployment."

/Why rejected:/ The "always set" assumption breaks the moment someone runs the
binary in a new context (local dev, a one-off script, a new deploy environment,
a debugging shell).  Load config at startup with proper error reporting that
names the missing variable.

***** ~OnceCell::get().unwrap()~ / ~LazyLock~ after init

/Argument for:/ "I initialized it earlier in ~main~."

/Why rejected:/ Usually a sign the data structure is wrong — ~LazyLock~ exposes
the value via ~Deref~ directly, and ~OnceLock~ has ~get_or_init~.  Restructure
rather than asserting initialization order at runtime.

**** How to apply

Treat ~.unwrap()~ and ~.expect()~ outside the test exemption as a defect.
Refactor to ~Result<_, E>~ and propagate with ~?~.  If the error type does not
compose, fix the error type (add a ~From~ impl, use ~anyhow~ / ~thiserror~,
etc.) — do not reach for ~.unwrap()~.

For literal-input cases (regex, addresses, URLs, etc.), use or write a macro
that validates at compile time.  The unwrap, if any, lives inside the macro on a
statically-proven-unreachable path — never in user code.

When reviewing diffs, flag every new ~.unwrap()~ / ~.expect()~ and check it
against the considered-exceptions list above.  "But this case is different" is
itself a yellow flag — usually it is one of the above wearing a different hat.

The bias to watch for: LLMs carry a strong "exceptions are friendly" instinct
from Python and other exception-driven languages where ad-hoc ~try/except~ is
normal. Rust's design rejects that worldview.  The discipline is: prove
unreachability to the compiler, or propagate.  There is no third option for
shipping code.

*** Rule of least power

Choose the least expressive construct that does the job.  The principle comes
from Tim Berners-Lee: a more powerful construct can do more, and that capacity
is a /liability/ for the reader, who has to verify what it actually does versus
what a narrower construct would have constrained it to. ~Option::map~ can only
transform-if-present; ~match~ on ~Option~ can do anything, so the reader has to
read both arms to be sure.  An iterator chain constrains the work to per-element
transformation and aggregation; a ~for~ loop with mutation can do anything at
all. The narrower form carries an invariant the broader one does not.

The Rust corollary is to prefer combinators and expressions over ~match~, ~if
let~, and ~for~ when the combinator form expresses the same control flow.
Concretely:

- ~Option::map~, ~Option::and_then~, ~Option::or_else~ over ~match~ on ~Option~.
- ~Result::map~, ~Result::map_err~, and ~?~ over ~match~ on ~Result~ for
  propagation.
- Iterator chains (~map~, ~filter~, ~filter_map~, ~find~, ~fold~, ~collect~)
  over ~for~ loops with mutation.
- ~let~ over ~let mut~; pure expression over statement-with-side-effects.
- ~Option::map_or~ / ~map_or_else~ over ~if let Some(x) = ... { ... } else { ...
  }~ when both arms are pure transforms.
- ~?~ over ~if cond { return Err(...) } else { ... }~ when the right shape
  exists. An early-return guard clause without an ~else~ is /already/ the
  least-powerful form for that flow — leave it alone.

The mechanical enforcement is a curated set of clippy lints in the ~manual_*~
family (~manual_map~, ~manual_filter~, ~manual_find~, ~manual_let_else~, etc.)
plus ~option_if_let_else~, ~map_unwrap_or~, ~question_mark~,
~single_match_else~, and friends.  These live under ~[workspace.lints.clippy]~
in the root ~Cargo.toml~ at ~warn~ — they have teeth without becoming a hard
block. When the imperative form is genuinely clearer (multi-line bodies, side
effects, nested control flow), override site-locally with
~#[allow(clippy::lint_name)]~ and a one-line comment explaining why.

The bias to watch for: LLMs reach for ~match~, ~for~, and early ~return~ because
that is how most training-set Rust reads, and because imperative constructs are
the most expressive available.  Maximum expressivity is the wrong default.
Reach for the narrowest construct that fits; escalate to ~match~ or a loop only
when the combinator form distorts the meaning.

*** Code Structure

**** Files and modules

~main.rs~ should only ever be an orchestrator.  Create sibling files for
associated functionality and have ~main.rs~ delegate to them.

Delegate obvious concerns: ~config.rs~ for configuration, ~logging.rs~ for
logging initialization.  It is okay if these wind up being a single struct and a
single function.

Keep ~.rs~ files under ~src~ for executables and ~lib~ for libraries.

**** Chains, local variables, and tap

Local variables should be generally frowned upon unless Rust specifically
requires them (for scope/ownership reasons), or a value has multiple uses.

You can further avoid local variables by using the ~tap~ crate to do logging in
the middle of a call chain without breaking the chain.

Another way to avoid local variables is to create structs atomically from
function calls.  The ~foo~ field can be assigned the result of the ~foo()~ call,
for example.  Prefer nouns (~foo()~) over verb-prefixed names (~get_foo()~),
since verbs are highly subjective in software.

Prefer functional combinators over explicit ~loop~ and early ~return~.  When
functions grow complicated, simplify by extracting smaller functions.

*** Debugging

When lacking information during debugging, permanent additions to the logs (at
appropriate levels) are one of the best cures.  The program should tell you what
ails it.

Create tests which exercise the problem you are running into.  Prefer mock
services with integration tests so we can test real behavior, over unit tests
which test contrived situations.

Tests and test scripts must exit with a proper error code.

*** Types as validation

Only use ~Option~ if the field is truly optional.  If a struct would be invalid
without a field set, do not use ~Option~.  Instead, make an intermediate type
that represents a candidate whose validity is not yet fully known.  This is very
helpful for processing phases.

For example, ~CliArgs~ can represent arguments as provided on the command line.
~ConfigFile~ can represent the configuration file's state.  Meshing the two
together yields a ~Config~ that represents a completely valid configuration.

* Persistent memory

Use the assistant's local memory only for facts specific to one clone or one
machine — local paths, machine quirks, transient personal context.  Anything
that is a project convention, decision, or standard belongs in the repository
(~llms.org~, ~CONTRIBUTING.org~, or ~docs/~), where every contributor benefits
from it and it can be reviewed.  Reaching for memory to record a convention is
the signal that it should be codified here instead.

* Capturing plans

When a planning pass produces a design worth keeping, record it as a ~TODO~
entry in ~tasks.org~ (the project's backlog), cross-linking related entries and
marking superseded ones, rather than leaving it only in an ephemeral plan file.

* Verification

** Use the Nix development shell when available

If ~nix~ is available (~which nix~ succeeds), run all commands inside the Nix
development shell rather than using whatever toolchain is on the system PATH:

#+begin_src sh :results none :exports code
nix develop --command just build
nix develop --command just test
#+end_src

This ensures the correct Rust toolchain and associated tools (such as ~rustfmt~)
are in use.  The pre-commit hook requires ~rustfmt~ in PATH, so ~git commit~
must also go through the shell:

#+begin_src sh :results none :exports code
nix develop --command git commit --message "your message"
#+end_src

** Run all tests after changes

Run ~just test~ after making any change to verify nothing is broken.
